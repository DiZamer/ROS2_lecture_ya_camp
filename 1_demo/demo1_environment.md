# Демонстрация: Готовность окружения

## Цель

Показать студентам 4 последовательные проверки: (1) `ros2 --help` — CLI существует и отвечает, (2) `ros2 run demo_nodes_cpp talker` — первый узел запущен и публикует сообщения в реальном времени, (3) `ros2 doctor --report` — диагностика окружения без ошибок, (4) `printenv RMW_IMPLEMENTATION` — видно, какой DDS используется (Fast DDS). Студент должен увидеть, что ROS2 — не абстракция, а реально запущенная система в контейнере.

## Подготовка до лекции

1. Контейнер уровня 2 открыт и готов (Dev Container пересобран).
2. Терминал открыт в VS Code, приглашение показывает `ubuntu@...`.
3. Команда `ros2 --help` проверена заранее.

## Контейнер

Dev Container уровня 2 — общий контейнер курса.

## Контекст для студентов

> «ROS2 не ставится на ваш компьютер. Все запускается в контейнере — изолированной среде с Ubuntu 24.04 и ROS2 Jazzy. У всех одинаковое окружение, независимо от вашего хоста — Windows, macOS или Linux. Давайте проверим, что ROS2 работает.»

## Что показать

### 1. Проверка CLI

```bash
ros2 --help
# Выводит: все подкоманды (run, topic, service, action, node, param, launch, ...)
```

**Что сказать**: «`ros2` — единая точка входа. Через нее вы запускаете узлы, проверяете topics, вызываете services, отправляете actions. Все управление ROS2 — через этот CLI.»

### 2. Первый запущенный узел

```bash
# ros2 run <пакет> <исполняемый_файл> — запуск узла из собранного пакета
# demo_nodes_cpp — стандартный пакет-демонстрация из ROS2 Jazzy
# talker — узел, который публикует std_msgs/String в топик /chatter раз в секунду
ros2 run demo_nodes_cpp talker
```

**Ожидаемый результат**:

```
[INFO] [talker]: Publishing: 'Hello World: 1'
[INFO] [talker]: Publishing: 'Hello World: 2'
[INFO] [talker]: Publishing: 'Hello World: 3'
```

**Что сказать**: «Мы только что запустили первый узел — `talker`. Он публикует сообщения в topic. Subscriber-а пока нет, но узел уже работает. Это программа, которая делает ровно одно дело: публикует строку раз в секунду.»

Остановить: `Ctrl+C`.

### 3. Диагностика окружения

```bash
# ros2 doctor — встроенная диагностика: проверяет RMW, discovery, clock sync, network
# --report — полный отчёт со статусом каждой проверки
ros2 doctor --report
```

**Ожидаемый результат**: таблица с проверками — все `OK` или `WARN`.

**Что сказать**: «`ros2 doctor` — диагностика окружения. Проверяет, что ROS2, зависимости и сеть настроены правильно. Если что-то не так — покажет предупреждение.»

### 4. Какой middleware используется

```bash
# Переменная окружения RMW_IMPLEMENTATION — выбор конкретной DDS-реализации
# Не установлена → используется дефолтная (Fast DDS в Jazzy)
printenv RMW_IMPLEMENTATION
```

**Ожидаемый результат**: `rmw_fastrtps_cpp`

**Что сказать**: «ROS2 использует DDS для передачи сообщений. Конкретная реализация — Fast DDS. Ее можно сменить на Cyclone DDS без изменения кода узлов, просто переключив переменную окружения. Подробнее — в статье про архитектуру ROS2.»

## Что сказать

- «То, что вы только что увидели — это demo-узел из пакета `demo_nodes_cpp`. Он написан на C++, но мы в курсе будем писать на Python — синтаксис проще, а ROS2 API одинаковый.»
- «Этот же узел мы можем запустить на Python: `ros2 run demo_nodes_py talker`. Поведение идентичное.»
- «Все практики в курсе выполняются в этом же контейнере. Никакой установки ROS2 на хост.»
- «Такой же контейнерный подход используется для виртуального робота TIAGo — там свой контейнер с ROS2 Humble, Gazebo и полной симуляцией.»

## Ожидаемый результат

- `ros2 --help` показывает справку с подкомандами
- `talker` печатает `Hello World` каждую секунду
- `ros2 doctor --report` показывает «All checks passed» или список проверок
- `printenv RMW_IMPLEMENTATION` показывает `rmw_fastrtps_cpp`

## Типичные проблемы

| Симптом | Причина | Исправление |
| --- | --- | --- |
| `bash: ros2: command not found` | Контейнер не запущен или собран без ROS2 | `Ctrl+Shift+P → Rebuild Container` |
| `demo_nodes_cpp` не найден | Пакет не установлен в образе | Проверить `Dockerfile` — должен быть `ros-jazzy-demo-nodes-cpp` |
| `rqt_graph` пустой или не запускается | Нет DISPLAY или X11-проброса | Проверить `devcontainer.json` → `containerEnv.DISPLAY` |

## План Б

Если контейнер не запускается или нет GUI для `rqt_graph`:

1. Показать скриншот работающего терминала с `ros2 --help` и `talker`.
2. Показать структуру `Dockerfile` и `devcontainer.json` — объяснить, что контейнер собирается из `osrf/ros:jazzy-desktop` и содержит все нужные пакеты.
3. Сказать: «На ваших компьютерах это будет работать точно так же — контейнер гарантирует одинаковое окружение.»
